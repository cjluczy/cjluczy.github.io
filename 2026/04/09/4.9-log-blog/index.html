GitHub Actions + Hexo 自动化发布踩坑记

背景

我想实现：通过 Dify 工作流自动抓取 GitHub 热门项目，生成 Markdown 文章，调用 GitHub API 推送到博客仓库，触发自动构建，更新 GitHub Pages。

理想很美好，现实很折磨。前后折腾了两天，踩了无数坑，记录如下。

问题一：第二次上传同名文件报 404

现象：第一次 PUT 成功，第二次 PUT 相同路径的文件时，API 返回 404。

原因：GitHub API 更新已存在文件时，必须在请求体中提供当前文件的 sha 值。

解决：

  • 方案 A（推荐）：每次生成不同的文件名，比如加上日期 github-hot-2026-04-09.md。
  • 方案 B：先 GET 获取 sha，再 PUT 带上 "sha": "xxx"。

问题二：API 推送成功，但博客没更新

现象：API 返回 201，但网站还是旧的。

原因：我直接往 master 分支推送了 Markdown 文件，但 GitHub Pages 需要的是静态 HTML 文件。Hexo 需要先构建。

解决：引入 GitHub Actions，监听 master 分支的推送，自动运行 hexo generate，然后把生成的 public/ 目录推送到 gh-pages 分支，GitHub Pages 从 gh-pages 分支读取。

问题三：子模块错误导致 Actions 失败

现象：Actions 报错 fatal: No url found for submodule path '.deploy_git' in .gitmodules，后来又报 themes/butterfly 子模块错误。

原因：之前用过 hexo deploy，留下了 .deploy_git 子模块；主题也可能被当作子模块添加过。

解决：

  • 删除 .gitmodules 文件。
  • 执行 git rm --cached themes/butterfly 移除子模块记录，然后重新添加为普通文件夹。
  • 在 workflow 中设置 submodules: false。

问题四：工作流配置混乱，部署分支写错

现象：Actions 显示成功，gh-pages 分支也有新文件，但 master/source/_posts 里没有新文章，且 gh-pages 分支出现的是一些奇怪的文件（如 test04）。

原因：我在 Dify 的 HTTP 请求节点中，把 "branch": "master" 改成了 "branch": "gh-pages"（因为之前让你改分支改晕了）。导致文章直接写到了 gh-pages 分支，而不是 master 分支的源码目录。Hexo 没有源码，自然不会生成任何东西。

解决：把 branch 改回 master。

问题五：Markdown 文件格式不对，Hexo 不识别

现象：master/source/_posts 里有文件，但 gh-pages 分支没有对应的 HTML。

原因：Dify 生成的 article 以 <think> 标签开头，然后才是 ---。Hexo 要求 Front Matter（---）必须是文件的第一个字符。

核心教训

  1. GitHub API 更新文件需要 sha，否则用不同的文件名。
  2. 源码分支（master）和部署分支（gh-pages）必须分开，不要在 API 调用时把分支写错。
  3. 子模块问题：删除 .gitmodules，用 git rm --cached 移除，workflow 里加 submodules: false。
  4. Hexo 文章的 Front Matter 必须从文件第一行开始，前面不能有任何字符。
  5. 不要同时混用多种部署方式（hexo deploy 和 peaceiris/actions-gh-pages 二选一）。
  6. 每次改配置只改一处，验证后再改下一处。改晕了就把分支名写错，浪费半天。

最后：自动化很美，但坑很多。保持配置简洁，分支职责清晰，能少走一半弯路。
